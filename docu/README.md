# LMS 프로젝트 수행계획서 (Spring Boot)

## 1. 프로젝트 개요

- **프로젝트명**: LMS (Learning Management System)
- **개발 목적**: 강의/수강/진도/과제/평가를 통합 관리할 수 있는 웹 기반 학습관리 시스템 구축
- **개발 방식**: 단계별 MVP 개발 → 기능 확장
- **프로젝트 경로**: `~/projects/lms`

---

## 2. 목표

1. 학습자/강사/관리자 역할 기반 기능 제공
2. 강의 콘텐츠 등록 및 수강 흐름(신청 → 학습 → 완료) 구현
3. 과제/퀴즈/성적 관리 기능 제공
4. 운영에 필요한 기본 관리자 기능(회원/강의/통계) 제공
5. 유지보수 가능한 Spring Boot 기반 아키텍처 확립

---

## 3. 범위

### 3.1 포함 범위 (In Scope)

- 회원가입/로그인/권한관리 (학습자, 강사, 관리자)
- 강의 생성/수정/삭제 및 커리큘럼 관리
- 수강 신청/취소 및 수강 이력
- 강의 진도(학습 완료율) 추적
- 과제 등록/제출/채점
- 퀴즈 응시/자동채점(객관식 중심)
- 공지사항 및 Q&A 게시판
- 관리자 대시보드(기초 통계)

### 3.2 제외 범위 (Out of Scope, 1차)

- 실시간 화상 강의(WebRTC)
- 결제/정산 시스템
- AI 기반 학습 추천 고도화
- 모바일 네이티브 앱(웹 반응형 우선)

---

## 4. 기술 스택 (초안)

- **Backend**: Java 21, Spring Boot 3.x, Spring Security, Spring Data JPA
- **Database**: PostgreSQL (개발 초기 H2 병행 가능)
- **Build**: Gradle
- **API 문서화**: Springdoc OpenAPI (Swagger)
- **인증 방식**: 세션 기반 또는 JWT (요구사항에 따라 결정)
- **테스트**: JUnit5, Mockito, Spring Boot Test
- **인프라(예정)**: Docker, Nginx, (선택) AWS/온프레미스

---

## 5. 시스템 아키텍처 (초안)

- **Layered Architecture**
  - Controller (API/요청 처리)
  - Service (비즈니스 로직)
  - Repository (DB 접근)
  - Domain(Entity)
- **권한 기반 접근 제어 (RBAC)**
  - ROLE_STUDENT, ROLE_INSTRUCTOR, ROLE_ADMIN
- **주요 도메인**
  - User, Course, Lecture, Enrollment, Assignment, Submission, Quiz, Question, Result, Notice

---

## 6. 개발 일정 (예시: 8주)

### Week 1: 기획/설계
- 요구사항 정리
- ERD 초안 작성
- API 명세 초안
- 프로젝트 초기 세팅

### Week 2: 인증/회원
- 회원가입/로그인
- 권한 모델 구현
- 사용자 프로필 관리

### Week 3: 강의 관리
- 강의 CRUD
- 커리큘럼(섹션/강의) 관리
- 강의 검색/목록

### Week 4: 수강/진도
- 수강신청/취소
- 수강 상태 및 진도율 추적

### Week 5: 과제
- 과제 생성/제출/조회
- 채점 및 피드백

### Week 6: 퀴즈/평가
- 퀴즈 생성/응시
- 자동채점 및 결과 저장

### Week 7: 관리자/게시판
- 공지사항/Q&A
- 관리자 통계 대시보드

### Week 8: 테스트/배포 준비
- 통합 테스트
- 성능 점검
- 배포 문서/운영 문서 정리

---

## 7. 품질 관리 계획

- **코드 컨벤션**: Google Java Style 기반 팀 컨벤션
- **브랜치 전략**: Git Flow 경량화 (`main`, `develop`, `feature/*`)
- **코드 리뷰**: PR 필수, 최소 1인 승인
- **테스트 목표**
  - Service 계층 단위 테스트 우선
  - 핵심 시나리오 통합 테스트

---

## 8. 리스크 및 대응

1. **요구사항 변경 빈도 증가**
   - 대응: 주간 단위 범위 고정, 변경 이력 관리
2. **인증/권한 복잡도 증가**
   - 대응: 초기 RBAC 단순화, 점진적 확장
3. **일정 지연**
   - 대응: MVP 우선순위 고정, 비핵심 기능 후순위
4. **데이터 모델 변경 비용**
   - 대응: 초기 ERD 리뷰 강화, 마이그레이션 체계화
